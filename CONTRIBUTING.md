# Contributing to duckdb-lttb

Thanks for your interest in contributing! This is a DuckDB extension implementing
LTTB downsampling. Before you start, set up your environment and read the
contribution workflow below.

## Environment setup

### Prerequisites

- **C++17 compiler** (clang recommended — matches CI)
- **CMake** ≥ 3.10
- **Ninja** and **ccache** for fast local rebuilds:
  ```sh
  brew install ninja ccache
  ```
- **DuckDB submodules** (already pinned at v1.5.2):
  ```sh
  git submodule update --init --recursive
  ```

### Format & static-analysis tools (one-time)

The format and tidy checks need pip-installed tools. Install them and add the
pip user bin directory to your `PATH`:

```sh
pip3 install "black>=24" cmake-format clang_format==11.0.1 clang-tidy
export PATH="$(python3 -c 'import site; print(site.getuserbase())')/bin:$PATH"
```

Add the `export PATH=...` line to your shell profile (`~/.zshrc` / `~/.bashrc`)
so it persists across sessions.

### Build

```sh
PATH="/opt/homebrew/opt/ccache/libexec:$PATH" GEN=ninja make
```

If an existing `build/release` was configured with Unix Makefiles, remove it
before switching generators:

```sh
rm -rf build/release
PATH="/opt/homebrew/opt/ccache/libexec:$PATH" GEN=ninja make
```

Build outputs:

```text
./build/release/duckdb                                          # DuckDB CLI with extension
./build/release/test/unittest                                   # test runner
./build/release/extension/lttb/lttb.duckdb_extension            # loadable extension
```

### Install the pre-commit hook

The repo ships a pre-commit hook that runs `make format-check` (the same check
CI runs). Install it once:

```sh
git config core.hooksPath .githooks
```

The hook auto-discovers the pip-installed tools, so it works even under Git's
sanitized hook `PATH`.

## Contribution workflow

1. **Open an issue** for anything non-trivial — bug reports, feature ideas,
   algorithm changes. Discuss before opening a PR for large work.
2. **Branch from `main`**: `git checkout -b my-fix`.
3. **Scope your change**: target ≤5 files, ≤200 lines per PR. One fix or
   feature = one PR.
4. **Verify before pushing** — run the gate sequence in order:
   ```sh
   make format-check
   make tidy-check
   ./build/release/test/unittest "test/sql/lttb.test"
   ```
   New SQL functions or behavior changes need SQLLogicTest cases in
   `test/sql/lttb.test`.
5. **Open a PR** against `main`. Fill in the PR template — including the
   review-confirmation checkbox.
6. **Address review feedback**. Push follow-up commits; avoid force-pushing
   during review unless asked.

## Code style

Enforced by `.clang-format` and `.clang-tidy` (all warnings are errors):

- ColumnLimit: 120, tab indentation (width 4)
- PointerAlignment: Right (`int *foo`)
- Classes/Functions: CamelCase; variables/members/parameters: lower_case
- Macros/static vars/enum constants: UPPER_CASE
- No `using namespace` in headers, no C-style casts, no commented-out code
- No raw `new`/`delete` — use DuckDB `unique_ptr`/`shared_ptr`
- Functions < 80 lines; extract helpers if larger

Run `make format` to auto-fix formatting. There is no auto-fix for clang-tidy —
fix warnings manually.

## Boundaries

- **Never** push to `main`, commit `build/`, or modify the `duckdb/` and
  `extension-ci-tools/` submodules.
- **Never** disable lint rules inline (no `NOLINT` — `WarningsAsErrors: '*'`).
- **Ask first** before changing the DuckDB version (v1.5.2), `CMakeLists.txt`,
  adding dependencies, or changing `.clang-format`/`.clang-tidy`.

## AI use policy

AI assistance is allowed, but contributors are responsible for every line of
code they submit. Specifically:

- **PR descriptions must be human-written.** Do not paste LLM output verbatim
  into the PR description.
- **You must understand and be able to explain every line** of your PR. The PR
  template includes a checkbox confirming this.
- **Rewrite AI-generated comments in your own words.** AI comments tend to
  over-explain and hedge — explain *why*, not *what*, and keep it short.
- Pure vibe-coded PRs (generated by AI with no human review or understanding)
  may be closed without review.

AI tooling and the `AGENTS.md` contract exist to help contributors produce
correct, standards-compliant code faster — not to replace human judgment.
